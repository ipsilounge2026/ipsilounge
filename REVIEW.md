# 기획서 vs 구현 비교/검토 가이드

## 목적

기획서에 명시된 요구사항이 실제 코드에 반영되었는지 정확하게 판별한다.
"구현 안 됨"으로 잘못 분류하는 오류를 방지한다.

---

## 검토 원칙

### 1. 기능 단위가 아니라 요구사항 단위로 비교한다

기획서의 항목명과 코드의 모듈/함수명이 1:1로 대응하지 않아도 된다.
기획서가 요구하는 **동작**이 어떤 방식으로든 충족되면 "구현 완료"다.

- **잘못된 예**: 기획서에 "관리자 검토 워크플로"가 있는데 `review_workflow.py`가 없으니 미구현
- **올바른 예**: `review_status` 필드 + `addendum` append-only 패턴으로 검토/피드백이 가능하니 구현 완료

### 2. 검토 순서

```
Step 1: 기획서 항목의 본질적 요구사항을 한 문장으로 정리
        → "관리자가 선배 기록을 검토하고 승인/수정요청할 수 있어야 한다"

Step 2: 해당 요구사항을 충족하는 기존 메커니즘이 있는지 확인
        → DB 모델 필드, API 엔드포인트, 프론트엔드 UI 모두 확인
        → 이름이 다르더라도 동작이 같으면 충족

Step 3: 기존 메커니즘으로 부족한 부분만 "미구현"으로 분류
        → 부분 충족이면 "부분 구현 — 부족한 점: ~~~"으로 표기
```

### 3. 설계 결정(Design Decision)을 미구현과 구분한다

의도적으로 빠진 것은 미구현이 아니다.

- 상담 기록 수정/삭제 불가 → 기획 의도 (immutable policy)
- 점수 보정 안 함 (소인수 과목) → 기획 의도 (표기만)
- 이런 항목을 "미구현"으로 올리지 않는다

### 4. 암묵적 충족 패턴을 인식한다

| 기획서 표현 | 코드에서 확인할 곳 |
|------------|-------------------|
| "공유 범위 설정" | `is_visible_to_user`, `is_visible_to_next_senior` 등 boolean 플래그 |
| "검토/승인 프로세스" | `review_status` + `addendum` 패턴 |
| "수정 불가" | UPDATE/DELETE 엔드포인트 부재 = 의도적 미제공 |
| "자동 채움" | `prefill_answers`, `source_survey_id`, `preserved_data` |
| "권한 분리" | `admin.role` 체크 + 라우터별 Depends 분기 |
| "이전 데이터 참조" | `source_survey_id`, 이전 세션 조회 로직 |

### 5. 판정 기준표

| 판정 | 기준 | 표기 |
|------|------|------|
| **구현 완료** | 요구사항의 핵심 동작이 모두 작동함 | ✅ |
| **부분 구현** | 핵심은 되지만 세부 사항이 빠짐 (UI 미연결 등) | ⚠️ + 부족한 점 명시 |
| **미구현** | 요구사항을 충족하는 코드가 전혀 없음 | ❌ |
| **설계 결정** | 의도적으로 제외된 항목 | 🔵 (미구현 아님) |

---

## 검토 보고서 작성 형식

```markdown
### [기획서 항목명]

**요구사항 요약**: (한 문장)
**판정**: ✅ / ⚠️ / ❌ / 🔵
**근거**:
- 모델: `파일명` — 해당 필드/관계
- API: `엔드포인트` — 해당 동작
- UI: `파일명` — 해당 화면 (있는 경우)
**부족한 점** (⚠️인 경우): 구체적으로 뭐가 빠졌는지
```

---

## 주의사항

- 키워드 매칭으로 판단하지 않는다 (이름이 달라도 동작이 같으면 충족)
- 기획서에 있는 모든 항목이 별도 모듈로 존재할 필요 없다
- "미구현" 판정 전에 반드시 기존 코드를 읽고 확인한다
- 확신이 없으면 "미확인"으로 두고 추가 조사 후 판정한다
